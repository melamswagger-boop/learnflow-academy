# Reflection

For this assessment, I chose to build an Education / Training Platform called **LearnFlow Academy** using React. I selected this domain because it fits naturally with the assignment requirements for multi-page navigation, asynchronous data handling, reusable components, forms, and dynamic user interaction. It also reflects a realistic frontend scenario where users need to discover information quickly, make decisions confidently, and track their own progress over time. Compared with some other domains, the education scenario also allowed me to create a clear and meaningful information hierarchy, which is important for both usability and professional frontend presentation.

From an architectural perspective, I wanted the project to remain modular and readable. I separated the codebase into pages, reusable components, context, services, hooks, data, and styles. This structure helped me avoid large monolithic files and made it easier to reason about where each responsibility belonged. For example, page files focus on assembling features, while smaller UI components such as `CourseCard`, `PageHeader`, `ProgressBar`, and `InputField` handle presentation in a reusable way. I used React Context to manage shared state for enrolments, progress, filters, and profile data. This approach felt appropriate for the size of the project because it keeps state centralised without introducing unnecessary complexity. If the application became much larger, a more advanced state management tool could be justified, but for this assessment I believed Context provided the best balance between simplicity and maintainability.

One important decision was to use mock JSON data with asynchronous service functions instead of a live API. This still allowed me to demonstrate industry-relevant frontend behaviour such as loading states, error states, and data-driven rendering, while reducing the risk of external API failures affecting the assessment. It also gave me more control over the user experience because I could shape the data to match the application features. The simulated delay in the service layer made the interface feel more realistic and helped satisfy the brief requirement for asynchronous data handling. In addition, using a small service layer separated data concerns from presentation logic, which improved readability and would make a future API integration easier.

The main challenges I faced were balancing feature completeness with code quality and making sure the user interface remained clear on both desktop and mobile screens. It can be tempting to add many features, but I focused on delivering the required features well: routing, filtering, enrolment, progress tracking, responsive layout, and form validation. Another challenge was maintaining consistency across different pages. I addressed this by creating a shared visual system with common spacing, button styles, cards, and page headers. This improved both usability and maintainability. Accessibility was another area I needed to think about carefully. I used semantic HTML, associated labels with form fields, preserved visible focus states, and ensured buttons and links were used appropriately so the application remained more inclusive and easier to navigate.

This project is relevant to industry practice because it reflects how frontend developers often work in component-based frameworks to build maintainable interfaces on top of existing backend services. The use of reusable components, clear state boundaries, responsive design, and accessibility considerations aligns with professional frontend standards. It also demonstrates the importance of giving users immediate feedback through loading indicators, error messages, progress updates, and confirmation messages after key actions. If I extended the project further, I would add persistent storage, automated tests, and integration with a real API. I would also consider analytics or recommendation features so the platform could provide more personalised learning pathways.

AI assistance was used only for grammar, structure, and planning support, and not for generating undeclared academic analysis or critical discussion.
